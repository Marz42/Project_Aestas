# 核心愿景与问题陈述

## 我们要解决的问题

科技、汽车、军事等垂直领域的专业编辑，每天面临严重的信息过载。他们需要在数十个新闻源中进行人工筛选，现有 RSS 工具只是“信息搬运工”，充斥着摘要截断、重复报道（几十家媒体发同一篇通稿）和无效噪音，导致编辑耗费大量精力“找线索”，而非“写深度”。

## 我们的解决方案

提供一个自动化的内容炼油厂：抓取高优信源 -> 智能提取全文 -> AI 自动去重并按编辑的专业视角（Prompt）提取核心线索 -> 最终生成带来源追溯的结构化简报，直接推送到编辑的工作群（如飞书/企微）。我们提供的是**信息增量和写作视角**，而不是简单的原文压缩。

## 为什么是现在

大语言模型的上下文窗口和逻辑能力大幅跃升，且 API 成本极低（甚至比抓取代理还便宜）。结合成熟的结构化输出技术（Structured Outputs），AI 已经可以完美替代人类完成“初级新闻线索梳理”的工作。

# 技术栈

> 抛弃 n8n 后，采用 **Python (FastAPI + Celery)** 是处理这类“爬虫+AI+定时任务”最稳健、生态最好的选择。前端采用现代化且极易上手的 **Vue 3 生态**。

## 前端 (管理后台：用于配置信源、查看新闻库、修改 Prompt)

| 层面 | 选型 | 版本 | 选型理由 |
|------|------|------|----------|
| 框架 | Vue 3 | 3.x | 组合式 API (Composition API) 逻辑复用好，国内生态极佳，开发 MVP 速度极快。 |
| 语言 | TypeScript | 5.x | 提供完善的类型提示，前后端数据结构对接更安全。 |
| 构建工具 | Vite | 5.x | 冷启动极快，热更新(HMR)体验完美，现代前端标配。 |
| UI 组件库 | Element Plus | 2.x | 极其适合做后台管理系统，开箱即用的表格、表单组件。 |
| 状态管理 | Pinia | 2.x | Vue 官方推荐，比 Vuex 更轻量，完美支持 TS。 |
| 路由 | Vue Router | 4.x | Vue 官方路由。 |
| CSS 方案 | Tailwind CSS | 3.x | 原子化 CSS，不用花时间想 class 名字，加速页面搭建。 |
| HTTP 客户端 | Axios | 1.x | 结合拦截器处理全局 Token 和错误提示。 |

## 后端 (核心大脑：处理爬虫任务、调度 AI、生成推送)

| 层面 | 选型 | 版本 | 选型理由 |
|------|------|------|----------|
| 框架 | FastAPI | 0.110+ | 现代、快速、原生支持异步 (Async)，自带 Swagger UI 接口文档，极度契合 AI 时代的 API 开发。 |
| 语言 | Python | 3.12+ | 爬虫与 AI 生态无可替代。 |
| ORM / 数据层 | SQLAlchemy 2.0 | 2.0+ | Python 届最稳健的 ORM，2.0 版本完美支持类型提示和异步。 |
| 缓存/中间件 | Redis | 7.x | 用于存储会话、防重复抓取的缓存、以及作为消息队列的 Broker。 |
| 任务调度 | Celery + Celery Beat | 5.x | **核心选型！** 处理“定时抓取”和“耗时的AI请求”。防止因网络波动导致 API 拥堵崩溃。 |
| AI 交互层 | Instructor / Pydantic | - | **强烈推荐**：不一定非要用 LangChain，用 `Instructor` 库配合 Pydantic，强制 LLM 输出 JSON 格式，极其稳健。 |
| 全文抓取层 | Jina Reader / Crawl4AI | - | Jina Reader (`r.jina.ai`) 直接将网页转为大模型友好的 Markdown，省去写手写解析逻辑。 |

## 数据库

| 数据库 | 用途 | 选型理由 |
|--------|------|----------|
| PostgreSQL | 主数据库 | 关系型结构（信源、文章、简报）的最佳选择。未来如需做向量检索，直接开启 `pgvector` 插件即可，无需引入外部向量库。 |
| Redis | 缓存 / 队列 | 充当 Celery 的 Broker 和 Backend，以及爬虫 URL 去重（布隆过滤器）。 |

# 演进规划

# 演进规划

## 短期

**目标**：跑通“采集 -> AI 结构化 -> 推送”的核心最小可用链路。

| 里程碑 | 关键交付物 | 预计完成 |
|--------|-----------|----------|
| M1: 基础设施搭建 | Docker 环境搭建，FastAPI + PostgreSQL + Celery 连通跑通第一个定时打印任务。 | 第 1 周 |
| M2: 采集与处理引擎 | 完成 RSS 抓取与 Jina Reader 网页清洗，接入 DeepSeek-V3 跑通单条新闻摘要生成并入库。 | 第 2-3 周 |
| M3: 简报与推送 | 完成按标签聚合逻辑，生成每日简报并成功推送到飞书机器人。 | 第 4 周 |

**当前聚焦**：跑通 M1 和 M2，先别管界面，先让后台代码能自动抓取并产生结构化好数据。

## 中期

**目标**：完善后台可视化，提升简报的“新闻价值”（去重、视角优化）。

| 里程碑 | 关键交付物 | 依赖 |
|--------|-----------|------|
| M4: 管理后台上线 | 基于 Vue 3 的后台，可可视化管理信源列表、查看处理日志、手动触发任务。 | 依赖 M3 跑通 |
| M5: 质量提升 | 引入文本聚类算法解决新闻重复问题；开放 Prompt 配置界面，让编辑自己调优关注点。 | 依赖 M4 |

**潜在风险**：优质信源的获取难度增加（部分网站封禁爬虫）；每日处理量上升导致 AI API Token 费用超标（需优化清洗逻辑）。

## 长期

**愿景**：从“工具”进化为编辑部的“AI 助理与选题资料库”。

| 方向 | 简述 | 触发条件 |
|------|------|----------|
| AI 深度洞察 | 针对某个热点新闻（如特斯拉降价），自动搜集全网不同视角的评论，生成多维度分析报告。 | 编辑反馈常规简报已无法满足深度要求 |
| 历史选题知识库 | 依托长期积累的清洗数据，引入向量检索，编辑输入“过去一年固态电池进展”，AI 即刻生成梳理。 | 数据库积累量达到 10万+ 级别 |
| SaaS 化试水 | 支持不同公司/媒体团队入驻，各自管理自己的信源、Prompt 和推送渠道。 | 现有编辑朋友强烈推荐给其他同行 |

# 风险与约束

## 已知风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| **AI 幻觉 (造假数据)** | 中 | 高 | 提示词中强制要求“必须完全依据原文提取，严禁脑补”；且推送卡片**必须附带原文链接**，编辑对最终发布负责。 |
| **目标网站反爬限制** | 高 | 中 | 避免高频并发抓取；使用 Jina Reader 等云端服务；实在不行换信源。 |
| **上下文超限 / 成本高** | 中 | 中 | 不要把几万字的原文直接丢给 LLM 生成每日简报。必须采用**“单条摘要 -> 摘要聚合生成简报”**的 Map-Reduce 两级架构。 |

## 约束条件

- **技术/资源约束**：由你业余时间全栈开发，因此必须克制需求，依赖优秀的开源框架（FastAPI/Vue）和云端 API 服务，尽可能采用成熟方案，绝不手写复杂的底层算法。
